# Elvern v1.0 Constructive Plan

This document is a repo-aware research and design pass based on the current Elvern tree as inspected on 2026-04-22.

It is intentionally practical:

- it separates what already exists in the repo from what is only proposed
- it focuses on the next feature wave, not a full product rewrite
- it assumes Elvern remains a private self-hosted media control plane with desktop VLC and external-player handoff as a core strength

Assumptions:

- references to current implementation are based on files present in this repo today
- VLC and Infuse are the only external players with explicit route/service support in the current codebase
- any mention of IINA, MPC-BE, or PotPlayer below is a v1.0 planning direction, not a claim that support already exists
- "Lite Playback" and "Full Playback" below are proposed product definitions built on top of the current browser/mobile playback infrastructure and Route 2 work

## 1. Movie Card / Cover UI

### Current repo observations

- `frontend/src/components/MediaCard.jsx` already keeps a clean poster-first card shape with a `2 / 3` poster area and a fixed lower body.
- `frontend/src/lib/movieTitles.js` already strips filename noise and produces a simplified display title for cards.
- The lower info area is currently split into:
  - title row: title plus source badge on the same horizontal line
  - meta row: duration, file size, year, and quality-rank chip
- `frontend/src/styles.css` makes `.media-card__copy` scrollable, but the title and the source badge still compete for width because `.media-card__title-row` uses a single horizontal flex row.
- This means long titles can wrap, but badges still steal line width from the most important text.

### Proposed v1.0 direction

- Keep the current poster ratio and overall poster-card feel.
- Treat the title block as the first owner of the lower card area:
  - simplified but full movie name shown first
  - up to 3 visible lines
  - if it still overflows, allow the title/copy area to scroll instead of letting badges cover or squeeze it
- Move all badges below the title block:
  - source/storage badge
  - rank chip
  - year, size, or any future storage marker
- Keep the lower region visually quiet so the card still reads like a movie poster rather than a dense metadata tile.

### Implementation phases

1. Separate the title block from the badge block in `MediaCard.jsx` and CSS.
2. Give the title area a stable 3-line layout budget before badge rows render below it.
3. Keep overflow inside the lower copy region only, with scroll as a last resort rather than hidden truncation.
4. Polish spacing and chip density for both desktop and phone widths.

### Risks / open questions

- The card already uses a simplified display title. v1.0 should decide whether the full canonical movie title needs a hover/title affordance elsewhere, or whether the card can stay simplified-only.
- "Storage" can mean different things in Elvern today: local DGX/local library, cloud source, or future reference label. The chip vocabulary should be settled before adding more badges.
- A 3-line title area needs careful height budgeting so cards still align cleanly in the grid.

### Recommended first implementation PR

- Frontend-only card layout PR:
  - `frontend/src/components/MediaCard.jsx`
  - `frontend/src/styles.css`
- Goal: title-first lower layout with badges below, no data-model changes.

## 2. Detail Page Info Modal

### Current repo observations

- `frontend/src/pages/DetailPage.jsx` already has a clear upper playback action area with:
  - `Open in VLC`
  - `Lite Playback`
  - `Full Playback`
  - iPhone external-app buttons where applicable
- The page also mixes non-playback actions into that same upper action cluster, including hide actions.
- Extra metadata currently lives inline below the player shell in a `detail-grid`, with sections such as:
  - Playback
  - Video
  - Audio & subtitles
  - Source file
- The page already has modal/backdrop patterns for reconnect and resume flows, so a blurred modal presentation style already exists in the repo language.

### Proposed v1.0 direction

- Keep playback actions in the upper playback area.
- Add a separate non-playback action row below the playback window for actions such as:
  - `Info`
  - `Hide for me`
  - `Hide for everyone` where allowed
- Move richer metadata into a dedicated modal/info card:
  - modal popup
  - blurred backdrop
  - structured info sections for playback, audio/subtitles, source, and technical details
- Keep the detail page itself more focused on watching and handoff actions.

### Implementation phases

1. Add an `Info` action below the playback window and move non-playback actions into that lower row.
2. Reuse the existing modal/backdrop visual pattern and render the current `detail-grid` inside the modal first.
3. Once stable, trim the inline metadata on the page so the detail view stays cleaner.

### Risks / open questions

- On smaller phone layouts, a full metadata modal may need a sheet-style presentation even if desktop uses a centered card.
- The current hide actions are already visible and useful. Moving them should not make moderation or personal hiding slower.
- If the modal becomes the only metadata access path, it needs obvious discoverability.

### Recommended first implementation PR

- Frontend-only detail-page modal PR:
  - `frontend/src/pages/DetailPage.jsx`
  - `frontend/src/styles.css`
- Goal: new `Info` modal below the player shell, while keeping playback buttons in the existing upper area.

## 3. Playback Strategy: Handoff vs Lite/Full Browser Playback

### Current repo observations

- `frontend/src/pages/DetailPage.jsx` already presents three distinct user-facing playback choices on supported devices:
  - `Open in VLC`
  - `Lite Playback`
  - `Full Playback`
- `backend/app/services/playback_service.py` already handles browser direct-play vs HLS fallback for the simpler `/api/playback/...` path.
- `backend/app/routes/browser_playback.py` and `backend/app/routes/mobile_playback.py` both sit on top of `backend/app/services/mobile_playback_service.py`.
- `mobile_playback_service.py` already distinguishes engine mode and playback mode:
  - `full` playback requires `engine_mode=route2`
  - Route 2 sessions expose fields such as `attach_revision`, `attach_ready`, `cache_ranges`, and active epoch state
- `docs/browser-playback-route2-cutover-checklist.md` shows that Route 2 is already being treated as a truth/authority model, not a cosmetic UI layer.
- Public docs are not aligned with that reality yet. Current repo docs still describe browser playback as "convenience" or "fallback only" in places like:
  - `docs/architecture.md`
  - `docs/setup.md`
  - `docs/operations.md`
  - `docs/ROADMAP.md`

### Proposed v1.0 direction

- Treat Elvern as a two-main-path playback product:
  - strong local/home/Wi-Fi path: VLC or external-player handoff is preferred
  - weak/remote/unstable path: browser playback plus caching is a first-class strategy
- Document browser playback as a real product path, not merely a backup.

Proposed mode definitions:

- `Open in VLC` / external handoff
  - preferred when the user is on the home network, same host, or otherwise has a strong local path to the source
  - maximizes subtitle/audio-track handling, local-player capability, and original-source playback
- `Lite Playback`
  - the quick-start browser path
  - intended to begin once roughly the first 45 seconds are ready
  - optimized for weak or unstable networks, quick resume, and smaller preparation windows
  - can still use direct browser playback when a file is trivially safe, but product-wise it should mean "low-friction watch now"
- `Full Playback`
  - the browser session intended to wait for a larger ready threshold, using current prepared range plus current web/network conditions
  - aims for longer continuous watching, stronger seek/recovery behavior, and smoother whole-movie playback with fewer jitters
  - should map naturally onto Route 2 authority, attach, and epoch lifecycle behavior

How current Route 2/cache work appears to fit:

- Route 2 already looks like the backbone for `Full Playback`
  - attach revisions
  - active/replacement epochs
  - readiness boundaries
  - cache range reporting
  - explicit recovery semantics
- The current browser/mobile manager already looks capable of supporting a product story where:
  - Lite is the quicker, smaller-prep path
  - Full is the more authoritative, more durable prepared-playback path

What appears already implemented:

- desktop VLC handoff and fallback contracts
- browser direct vs HLS decisioning
- browser/mobile session creation
- Route 2 session/epoch lifecycle foundations
- cache/range reporting and attach-readiness semantics

What appears partial:

- product-level explanation of when users should choose Lite vs Full
- consistent naming and copy across docs/UI
- an explicit auto-strategy layer that chooses handoff vs Lite vs Full based on network/path confidence

What appears missing:

- a single documented playback policy for desktop browser, mobile browser, and external-player entrypoints
- v1.0 wording that promotes browser playback to first-class weak-network strategy
- explicit user-visible guidance for when Lite vs Full is recommended

### Implementation phases

1. Naming and documentation pass:
   - align README/docs/UI copy to the two-main-path model
   - define Lite and Full clearly
2. Decision-surface pass:
   - improve page-level recommendations and wording without changing the core engine yet
3. Policy pass:
   - later add an actual recommendation layer for strong-local vs weak-remote path selection
4. Engine hardening pass:
   - continue Route 2 stabilization and cache/recovery work under the Full Playback meaning

### Risks / open questions

- It is not yet clear whether non-mobile desktop browsers should eventually use the Route 2 browser session path by default, or keep a simpler split between `/api/playback` and `/api/browser-playback`.
- The product definition of Lite should avoid collapsing into "random fallback name for anything that is not Full."
- Direct MP4 browser play should remain possible, but the user-facing language should decide whether that is presented as part of Lite or as a transparent fast-path inside browser playback.

### Recommended first implementation PR

- Documentation and copy alignment PR:
  - document strong-network handoff vs weak-network browser strategy
  - define Lite and Full in user-facing language
  - do not change transport logic yet

## 4. External Player Matrix and Best-Effort Detection

### Current repo observations

- Desktop helper readiness and releases already exist in the repo:
  - `/api/desktop-helper/status`
  - `/api/desktop-helper/verify`
  - `/api/desktop-helper/release...`
- `backend/app/services/desktop_helper_service.py` already stores:
  - device/browser seen state
  - helper version/platform/arch
  - VLC detection state/path
  - helper verification callbacks
- Linux same-host does not require the helper; the current helper readiness flow already treats Linux differently.
- `backend/app/routes/native_playback.py` only supports iOS external launch targets for:
  - `vlc`
  - `infuse`
- `backend/app/services/transport_controller_service.py` only defines explicit player capability profiles for:
  - `vlc`
  - `infuse`

### Proposed v1.0 direction

- Keep detection lightweight for now.
- Path 1 only:
  - browser-side best-effort detection
  - platform filtering
  - protocol test where possible
  - helper verify callback where available
  - stored readiness state that can be downgraded when signals go stale or fail
- Do not build a heavy native-detection system yet.

Proposed player/platform matrix:

| Player | Current repo support | Likely platforms | v1.0 direction |
| --- | --- | --- | --- |
| VLC | Yes on desktop handoff; yes on iOS external route | Linux, Windows, macOS, iOS, Android | Core first-class player |
| Infuse | Yes on iOS external route only | iOS, iPadOS, maybe macOS later | Keep as Apple-platform external-player option |
| IINA | No repo support yet | macOS | Candidate only after protocol/launch spike validates a practical path |
| MPC-BE | No repo support yet | Windows | Candidate only after Windows launch/protocol spike validation |
| PotPlayer | No repo support yet | Windows | Include in planning if helper-based or protocol launch proves feasible |

Install/show-hide rules for v1.0:

- Linux same-host:
  - do not push helper install as required
  - show VLC readiness only
- Windows/macOS:
  - show helper download/setup guidance only on supported desktop platforms
  - show player options that actually make sense for that platform
- iPhone/iPad:
  - surface VLC and Infuse only
  - do not show desktop-helper install UX
- Unsupported or unvalidated player/platform combos:
  - hide by default rather than teasing future support as if it already exists

Readiness-state rule:

- Previously installed but now removed apps should stop being treated as available after a failed protocol attempt, failed verify callback, or stale readiness timeout.
- Stored readiness should be treated as advisory, not as proof.

### Implementation phases

1. Define the user-facing matrix and platform filters.
2. Extend readiness UI to distinguish:
   - supported on this platform
   - not installed / not verified
   - previously seen but now stale
3. Add best-effort downgrade rules when a protocol test or helper verification stops succeeding.
4. Only after that, run narrow validation spikes for IINA, MPC-BE, and PotPlayer.

### Risks / open questions

- This repo does not currently prove stable protocol or helper-launch behavior for IINA, MPC-BE, or PotPlayer.
- Best-effort detection can easily overclaim certainty. v1.0 should keep wording like "ready", "not verified", "last confirmed", and "may be unavailable now."
- PotPlayer should stay in the matrix because it is a realistic Windows player request, but it should remain clearly marked as unvalidated until a narrow spike proves a real launch path.

### Recommended first implementation PR

- Platform/player readiness UX PR:
  - extend current helper/install surface to show per-platform supported players and readiness wording
  - keep backend changes minimal and reuse current status/verify plumbing

## 5. Search Normalization

### Current repo observations

- Library search currently runs in Python after loading rows from the DB, using:
  - `backend/app/services/library_service.py`
  - `backend/app/services/title_normalization.py`
- The current normalization already does useful work:
  - lowercase
  - punctuation stripping
  - article dropping
  - tokenization
  - some acronym alias generation
- The current matcher compares tokenized forms and normalized phrases, but it does not maintain a compact no-space comparison key.
- That means a query like `Who Am I` can match `who am i`, while a compact query like `whoami` can still miss because:
  - indexed tokens are `who`, `am`, `i`
  - normalized phrase contains spaces
  - the search index does not keep a `whoami` compact variant

### Proposed v1.0 direction

- Keep the current search-index style, but extend it with compact-form matching:
  - lowercase
  - remove punctuation
  - collapse spaces
  - compare compact forms with spaces removed
- Treat compact matching as a search feature, not as a title rewrite feature.
- If search scale later requires it, add persisted normalized search keys or computed columns, but do not start there unless the current in-memory search path becomes too slow.

### Implementation phases

1. Extend `build_search_index` and `match_search_query` with compact-form comparisons.
2. Add regression tests for:
   - `who am i`
   - `whoami`
   - punctuation variants
   - apostrophe variants
3. Only later, if needed, introduce a DB-backed normalized title/search key for faster large-library matching.

### Risks / open questions

- Compact matching increases collision risk for short titles and abbreviations.
- Non-English and non-Latin titles may need a later normalization pass beyond the current ASCII-oriented approach.
- Search behavior should stay predictable; overly fuzzy matching can make results feel noisy.

### Recommended first implementation PR

- Backend-only search-normalization PR:
  - extend current in-memory search index with compact keys
  - add targeted tests
  - avoid schema changes in the first pass

## 6. Library Freshness Check / Auto-Scan

### Current repo observations

- `backend/app/main.py` still enqueues a scan on startup whenever `ELVERN_SCAN_ON_STARTUP` is true.
- `backend/app/services/scan_service.py` already provides useful single-flight behavior:
  - one scan at a time
  - thread/lock ownership
  - state tracking
- `backend/app/media_scan.py` already has a decent full-scan core:
  - recursive walk under `ELVERN_MEDIA_ROOT`
  - compare `file_size` and `file_mtime`
  - skip unchanged local files
  - ffprobe only changed files
  - delete removed local rows
- Today there is still no lightweight pre-check before startup scan:
  - no directory snapshot
  - no media-root fingerprint
  - no watcher
  - no cheap "likely changed?" gate before the full walk

### Proposed v1.0 direction

- Stop treating full startup scan as the default first reaction.
- Introduce a lightweight freshness-check layer first, then run a full scan only when changes are likely.

Recommended phases:

- Initial cheap startup/login check
  - compare a persisted lightweight snapshot against current root state
  - use top-level directory mtimes, counts, and media-root identity/fingerprint
  - if likely unchanged, do not enqueue a full scan
- Optional background watcher
  - best-effort only
  - useful on local Linux filesystems
  - advisory on NAS/Docker/network mounts where events may be incomplete or unreliable
- Manual rescan remains available
- Debounce and single-flight scan protection stay in place

Suggested snapshot signals:

- media root path/fingerprint
- top-level directory names and mtimes
- rolling file count and byte totals where cheap
- last completed scan summary
- optional sample hashes only if proven cheap enough

### Implementation phases

1. Add a lightweight scan-freshness snapshot store and startup check.
2. Change startup behavior from "always scan when enabled" to "cheap check, then enqueue full scan only when likely dirty."
3. Add optional periodic/background watcher support behind a feature flag.
4. Preserve manual rescan and existing single-flight protection.

### Risks / open questions

- Network mounts and Docker bind mounts can make filesystem events unreliable, so watcher-based accuracy cannot be assumed.
- A cheap snapshot can still miss edge cases if only deep nested files change and parent mtimes do not bubble as expected on a given filesystem.
- Cloud-library syncing is separate today; local-library freshness work should not silently redefine cloud sync policy.

### Recommended first implementation PR

- Backend freshness-gate PR:
  - add cheap snapshot/predicate logic ahead of startup scan
  - keep the existing full scan worker intact
  - do not change the media scan algorithm yet

## 7. Google Drive / Cloud Accounts / Multi-User

### Current repo observations

- Current behavior is not admin-only. It is mixed, with clear per-user pieces:
  - Google OAuth tokens are stored per `user_id` in `google_drive_accounts`
  - any connected user can add personal cloud libraries
  - admins can create shared libraries
  - users can hide shared libraries for their own account
- Relevant current files include:
  - `backend/app/routes/cloud_libraries.py`
  - `backend/app/services/cloud_library_source_service.py`
  - `backend/app/services/cloud_provider_auth_service.py`
  - `frontend/src/pages/SettingsPage.jsx`
- `SettingsPage.jsx` already exposes:
  - `My Libraries`
  - `Shared Libraries`
  - `Google Drive Setup`
- There is already useful in-app guidance, but there does not appear to be a simple user-facing standalone doc that explains:
  - how a normal user connects their own Google account
  - what "My Libraries" vs "Shared Libraries" means
  - what breaks if the owner account disconnects

### Proposed v1.0 direction

- Keep the current mixed model, but document and harden it more clearly:
  - per-user Google account connection is the base model
  - admin-managed shared source remains valid
  - user hide/show visibility stays per user
- Clarify ownership and token-security rules:
  - personal libraries belong to the user who connected the account
  - shared libraries still depend on the owner/admin account token unless later redesigned
  - tokens stay server-side only
  - reconnect and revoke flows should remain tied to the owning user account

Current gap summary:

- good UI affordance exists in Settings
- public docs are thin for regular user onboarding
- ownership/failure semantics are not explained well enough for multi-user family use

### Implementation phases

1. Documentation pass for normal users and admins:
   - how to connect a personal Google account
   - what shared libraries mean
   - what "hide for me" does
2. UI clarity pass:
   - show clearer ownership/reconnect implications on shared sources
3. Later policy pass:
   - decide whether long-term shared sources should remain tied to one user's token or move toward a more explicit admin-managed account model

### Risks / open questions

- Shared libraries owned by an admin account still depend on that account's refresh token. If that account disconnects, every user can be affected.
- v1.0 should decide whether "shared source" is intentionally "shared from an owner's connected account" or whether it wants a more explicit server-managed provider identity later.
- Cloud-source permission boundaries for future non-Google providers may not map exactly to the current model.

### Recommended first implementation PR

- Docs and settings-clarity PR:
  - explain personal vs shared cloud libraries using the current model
  - avoid auth/token redesign in the first pass

## 8. Settings: Media Library Reference

### Current repo observations

- Settings already has an admin-only `Poster reference location` section:
  - frontend: `frontend/src/pages/SettingsPage.jsx`
  - backend: `backend/app/routes/admin.py`
  - app-setting owner: `backend/app/services/app_settings_service.py`
- There is no current `Media Library Reference` setting.
- Elvern already exposes rough source distinctions in the UI such as:
  - `DGX`
  - `Cloud`
  - source labels on detail/cards
- There is already a difference between:
  - actual media storage/path config (`ELVERN_MEDIA_ROOT`, platform roots)
  - user-facing source labels

### Proposed v1.0 direction

- Add a `Media Library Reference` section above `Poster reference location`.
- Keep it intentionally lightweight in v1:
  - label/reference purpose only
  - no change to scan root, platform mapping, or path semantics
- Intended use:
  - local movie storage location/reference label
  - clarify what "DGX" means in a given deployment
  - distinguish local library identity from cloud libraries

Recommended v1 shape:

- one admin-managed reference label
- optional descriptive note or location hint
- visible where current local-source labels would otherwise feel too generic

### Implementation phases

1. Add the setting as a label/reference-only admin setting.
2. Surface it above Poster Reference in Settings.
3. Later, optionally replace or supplement generic local-source labels in detail/cards with that reference label.

### Risks / open questions

- This should not turn into a second path-configuration system. Actual media-root and platform-root config should stay where they are.
- If users want multiple local storage labels later, that becomes a different problem than the simple v1 reference label.
- The exact label vocabulary needs a product decision:
  - `DGX`
  - `Local Library`
  - user-supplied custom label

### Recommended first implementation PR

- Small settings/admin PR:
  - add a label-only Media Library Reference above Poster Reference
  - do not change storage/path behavior

## 9. Documentation Updates Needed Later

### Current repo observations

- The main public docs have now had an initial playback-language alignment pass, but future doc edits should keep that wording synchronized as playback behavior evolves.
- The sections most likely to need later touch-ups are:
  - `README.md`
  - `docs/setup.md`
  - `docs/operations.md`
  - `docs/architecture.md`
  - `docs/ROADMAP.md`

Historical wording that prompted this alignment pass:

- `docs/architecture.md`
  - language that treated browser playback as convenience-only
  - language that described browser playback as secondary
- `docs/setup.md`
  - language that reduced browser playback to a convenience-only mode
- `docs/operations.md`
  - language that framed browser playback mainly as fallback
- `docs/ROADMAP.md`
  - language that grouped browser playback under convenience/fallback wording

### Proposed v1.0 direction

- After playback terminology is settled, update public docs to describe:
  - handoff/VLC as the preferred strong-network path
  - browser playback plus caching as the first-class weak-network path
  - Lite and Full as product-level browser modes
- Keep the wording modest. The docs should say what Elvern is trying to optimize for, not pretend automatic transport selection is already finished.

### Implementation phases

1. Update the architecture and setup docs after Lite/Full terminology is finalized.
2. Update README once the first UI/copy pass lands, so public positioning matches the product surface.
3. Update operations/troubleshooting text after the first playback-policy pass is live.

### Risks / open questions

- If docs are updated before the UI and behavior names settle, they will drift again quickly.
- The docs should not overpromise remote-network intelligence that does not exist yet.

### Recommended first implementation PR

- Documentation alignment PR after the first playback-language and settings/info UX slices are accepted.
